# 业务、单车 通道问题

## 1. 多车通道和单车通道的差异？  
多车通道： 主要通过订阅业务实现对多车的订阅监听；  
单车通道： 直接订阅单车状态。


## 2. 订阅多车之后，其他针对单车的SDK释放会影响订阅的进程吗？

业务通道和单车通道是不同的两个通道，互相之前没有影响；  
如果你单车通道想连其他的车，必须先销毁之前的单车连接；  
业务要切换业务也是必须先销毁之前的。

## 3. 什么时间用业务通道，什么时间用单车通道？

监听多车实时状态，一般可以用于多车状态监测；  
如果想对车控制或者发起指令、任务等，还是需要连接单车通道。

## 4. 订阅多车通道，有时需要传入sourceType，sourceType是什么，又该怎么传？  

sourceType 是指订阅来源，autoxing要求所有二开项目必须传6（数字类型），否则会影响二开公司的统计数据。

## 5. 多车通道有很多入口不建议使用，为什么？

多车通道有很多入口不建议使用，主要有以下几个原因：  
1）难度系数高，对开发者要求高，需要足够了解多车通道的接口，才能使用；  
2）多车通道的接口，业务需求会相对复杂，担心项目不闭合导致不必要的问题；  
3）多车通道不是普遍需求，一般单车通道就已足够满足80%的业务需求；  
4）多车接口对底盘服务有版本要求，不适配所有车辆。

## 6. 多车通道可以订阅任务吗？  

没有专门多车监听子任务的入口，只有监听多车状态的入口，但是机器人实时状态会返回机器人当前的任务状态；  
如果要监听某个任务可以订阅任务状态。
